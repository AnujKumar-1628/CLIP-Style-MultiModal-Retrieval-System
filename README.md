# CLIP-Style MultiModal Retrieval System

An end-to-end image-text retrieval system inspired by CLIP. The project trains a dual-encoder model on paired images and captions, evaluates retrieval quality with Recall@K/MRR, builds searchable vector indices, and serves text-to-image search through a FastAPI API.

## Current Results

Latest test-set retrieval performance using `artifacts/checkpoints/clip_best.pt`:

| Direction | R@1 | R@5 | R@10 | MRR | Median Rank |
| --- | ---: | ---: | ---: | ---: | ---: |
| Text -> Image | 16.4% | 40.9% | 54.1% | 0.284 | 9 |
| Image -> Text | 20.8% | 48.8% | 60.8% | 0.336 | 6 |

## Project Overview

This repository covers the full retrieval lifecycle:

1. Prepare Flickr30k-style image-caption data.
2. Train a CLIP-style contrastive model.
3. Evaluate text-to-image and image-to-text retrieval.
4. Build embedding artifacts and vector indices.
5. Serve retrieval with a local API.

The current model stack is:

| Component | Implementation |
| --- | --- |
| Image encoder | ResNet-50 from `torchvision` |
| Text encoder | DistilBERT from `transformers` |
| Projection heads | MLP projection to shared embedding space |
| Loss | Symmetric InfoNCE / CLIP contrastive loss |
| Index backend | Flat NumPy index by default, FAISS-supported options available |
| API | FastAPI + Uvicorn |

## Data Setup

The default dataset config is [configs/data.yaml](configs/data.yaml).

Expected raw data layout:

```text
/content/local_data/data/raw/
├── captions.txt
└── flickr30k_images/
    ├── image_1.jpg
    ├── image_2.jpg
    └── ...
```

The captions file must contain image names and captions. The loader supports canonical columns:

```text
image_name, caption_id, caption
```

It also supports these aliases:

```text
comment_number -> caption_id
comment        -> caption
```

If running locally, update these fields in [configs/data.yaml](configs/data.yaml):

```yaml
dataset:
  captions_file: path/to/captions.txt
  images_dir: path/to/flickr30k_images
```

Splits are generated by image ID to avoid caption/image leakage:

```yaml
split:
  train_ratio: 0.92
  val_ratio: 0.04
  test_ratio: 0.04
  output_dir: data/splits
```

## Training

Run:

```bash
./scripts/train.sh
```

Equivalent explicit command:

```bash
./scripts/train.sh configs/training.yaml configs/model.yaml configs/data.yaml
```

Training outputs:

```text
artifacts/checkpoints/
├── clip_best.pt
├── clip_last.pt
└── clip_epoch_XXXX.pt
```

Important training config files:

| Config | Purpose |
| --- | --- |
| [configs/training.yaml](configs/training.yaml) | LR, scheduler, epochs, checkpoints, early stopping |
| [configs/model.yaml](configs/model.yaml) | Encoders, projection heads, logit scale, loss |
| [configs/data.yaml](configs/data.yaml) | Dataset paths, image/text preprocessing, dataloaders |

The default run uses:

```yaml
optimizer:
  lr: 0.00008

runtime:
  epochs: 15
  use_amp: true
  grad_accumulation_steps: 2

checkpoint:
  monitor: val/loss
  dir: artifacts/checkpoints
```

## Evaluation

Run retrieval evaluation:

```bash
./scripts/evaluate.sh
```

Explicit form:

```bash
./scripts/evaluate.sh configs/retrieval.yaml configs/model.yaml configs/data.yaml artifacts/checkpoints/clip_best.pt
```

[configs/retrieval.yaml](configs/retrieval.yaml) contains the checkpoint path:

```yaml
checkpoint:
  path: artifacts/checkpoints/clip_best.pt
  strict: true
  state_dict_key: model_state_dict
```

Evaluation reports:

```text
artifacts/metadata/retrieval_eval.json
```

Metrics include:

| Metric | Meaning |
| --- | --- |
| `r@1` | Correct item appears at rank 1 |
| `r@5` | Correct item appears in top 5 |
| `r@10` | Correct item appears in top 10 |
| `mrr` | Mean reciprocal rank |
| `mean_rank` | Average rank of first correct match |
| `median_rank` | Median rank of first correct match |

Example successful baseline from this project:

```text
text_to_image/r@1  = 0.164
text_to_image/r@5  = 0.409
text_to_image/r@10 = 0.541

image_to_text/r@1  = 0.208
image_to_text/r@5  = 0.488
image_to_text/r@10 = 0.608
```

If metrics are near zero, first verify that the trained checkpoint is being loaded.

## Build Retrieval Index

Before serving search, build embeddings and vector indices:

```bash
./scripts/build_index.sh
```

Explicit form:

```bash
./scripts/build_index.sh configs/retrieval.yaml configs/model.yaml configs/data.yaml test
```

This builds:

```text
artifacts/embeddings/
├── image_corpus.npy
└── text_corpus.npy

artifacts/indices/
├── image_index.bin
└── text_index.bin

artifacts/metadata/
├── image_corpus_ids.json
├── image_corpus_meta.json
├── image_index_index_meta.json
├── text_corpus_ids.json
├── text_corpus_meta.json
└── text_index_index_meta.json
```

The indexing path loads the same trained checkpoint configured in [configs/retrieval.yaml](configs/retrieval.yaml), so index embeddings and API query embeddings use the same model weights.

## Run API

Start the API:

```bash
./scripts/run_api.sh
```

On Windows/Git Bash, the script automatically maps default `0.0.0.0` to `127.0.0.1` for local serving.

Explicit command:

```bash
./scripts/run_api.sh 127.0.0.1 8001
```

Open:

```text
http://127.0.0.1:8001/docs
```

Available endpoints:

| Endpoint | Method | Purpose |
| --- | --- | --- |
| `/` | GET | API status |
| `/health` | GET | Health check |
| `/search/text-to-image` | POST | Search images from a text query |

Example request body for `/search/text-to-image`:

```json
{
  "query": "a dog running through grass",
  "top_k": 5
}
```

Example response:

```json
{
  "hits": [
    {
      "rank": 1,
      "item_id": "123456789.jpg",
      "score": 0.31,
      "metadata": {}
    }
  ]
}
```

## Configuration Guide

### Data

Edit [configs/data.yaml](configs/data.yaml) for:

```yaml
dataset:
  captions_file: ...
  images_dir: ...

loader:
  batch_size: 256
  num_workers: 8
```

### Model

Edit [configs/model.yaml](configs/model.yaml) for:

```yaml
image_encoder:
  architecture: resnet50

text_encoder:
  architecture: distilbert-base-uncased

projection_heads:
  embed_dim: 256

loss:
  name: symmetric_info_nce
```

### Training

Edit [configs/training.yaml](configs/training.yaml) for:

```yaml
runtime:
  epochs: 15

optimizer:
  lr: 0.00008
```

For stronger results, good next experiments are:

```yaml
runtime:
  epochs: 30
```

```yaml
projection_heads:
  image:
    dropout: 0.1
  text:
    dropout: 0.1

loss:
  label_smoothing: 0.0
```

### Retrieval

Edit [configs/retrieval.yaml](configs/retrieval.yaml) for:

```yaml
checkpoint:
  path: artifacts/checkpoints/clip_best.pt

runtime:
  split: test
  ks: [1, 5, 10]

search:
  default_top_k: 10
```

### API

Edit [configs/api.yaml](configs/api.yaml) for:

```yaml
server:
  host: "0.0.0.0"
  port: 8001

paths:
  retrieval_config: "configs/retrieval.yaml"
  model_config: "configs/model.yaml"
  data_config: "configs/data.yaml"
```

## Troubleshooting

### `ModuleNotFoundError: No module named 'src'`

Run scripts from the repo root:

```bash
cd /path/to/CLIP-Style-MultiModal-Retrieval-System
./scripts/evaluate.sh
```

The scripts export `PYTHONPATH` automatically. You can also install the project:

```bash
pip install -e .
```

### Evaluation metrics are almost zero

Check that [configs/retrieval.yaml](configs/retrieval.yaml) points to a trained checkpoint:

```yaml
checkpoint:
  path: artifacts/checkpoints/clip_best.pt
```

Also confirm the file exists:

```bash
ls artifacts/checkpoints
```

### API fails with `getaddrinfo failed`

Use localhost explicitly:

```bash
./scripts/run_api.sh 127.0.0.1 8001
```

Then open:

```text
http://127.0.0.1:8001/docs
```

### `/favicon.ico` returns 404

This is harmless. Browsers request a favicon automatically; the API does not define one.

### Search endpoint returns 500 or 503

Build the index first:

```bash
./scripts/build_index.sh
```

Then restart the API:

```bash
./scripts/run_api.sh 127.0.0.1 8001
```

### API returns filenames but empty metadata

This is expected with the current metadata pipeline:

```json
"metadata": {}
```

The backend is returning ranked image IDs and scores. Add richer metadata during embedding/index creation if you want captions, local image paths, or preview URLs in API responses.

## Notes on Results

For the current architecture and training budget, the project has reached a usable baseline:

```text
Text -> Image R@10: about 54%
Image -> Text R@10: about 61%
```

This is suitable for a project/demo baseline. To push accuracy higher, prioritize longer training, lighter projection dropout, no label smoothing, and eventually a stronger image encoder such as a ViT backbone.
